PORTFOLIO PROJECT · SYSTEM DESIGN

PayCore

A distributed payment processing platform demonstrating fintech backend architecture — wallet transfers, double-entry ledger, idempotency, atomicity, and settlement simulation.

Java · Spring Boot PostgreSQL · Primary DB Redis · Idempotency Cache Docker · Containerized REST · API Style Maven · Build
01 · Overview

What is PayCore?

PayCore simulates how money moves inside a payment company. It is NOT a real payment gateway. The focus is on correctness — money cannot be created or destroyed, every movement is recorded in a ledger, every API is idempotent, and every wallet operation is atomic.

🏦
Wallet-to-Wallet Transfer
Single payment mode. User initiates transfer. Money moves from source wallet to destination wallet. Simple, clean, focused.
📒
Double-Entry Ledger
Every transfer creates two ledger entries. Every debit has a matching credit. Total debits always equal total credits.
🔑
Idempotency
Client sends an idempotency key. If the same request is retried, the original result is returned. Money is never moved twice.
⚛️
Atomicity
All DB writes for a payment (debit + credit + ledger + payment record) happen in a single PostgreSQL transaction. All or nothing.
🔁
Refunds
Refund reverses the original ledger entries. Merchant wallet is debited, user wallet is credited. New ledger entries are created.
🕐
Settlement Simulation
Funds are held by platform on payment. A scheduler releases them to the merchant on T+1 or T+2 configurable delay.

03 · Service Boundaries

Modules (Single Deployable, Logical Separation)

This is a modular monolith — one Spring Boot app with clear package-level separation. Not microservices. This is intentional: simpler to run, easier to demonstrate, and the boundaries are still clean.

User Module
Register users, fetch user info, manage user identity. Each user owns exactly one wallet.
POST /users GET /users/:id
Merchant Module
Merchant onboarding, activation/deactivation, merchant profile. Each merchant owns exactly one wallet.
POST /merchants PATCH /merchants/:id/status
Wallet Module
Manage balances. Credit and debit operations. Balance is the live state; ledger is the source of truth. Supports pessimistic locking.
GET /wallets/:id GET /wallets/:id/balance
Payment Module
Core of the system. Create payments, process them, track status. Handles the entire payment lifecycle including refunds.
POST /payments GET /payments/:id POST /payments/:id/refund
Ledger Module
Immutable record of all financial movements. Double-entry accounting. Never updated, only appended. Source of truth for reconciliation.
GET /ledger/:accountId GET /ledger/reconcile
Settlement Module
Scheduled job releases held funds to merchants. Configurable T+1 or T+2 settlement mode. Runs as a Spring @Scheduled task.
Scheduled Job GET /settlements
04 · Core Payment Flow

User → System → End (Step by Step)

1

Client Sends Payment Request

User hits POST /payments with amount, source wallet, destination wallet, and an idempotency key in the header.

Header: Idempotency-Key: <uuid>
2

Idempotency Check (Redis)

System looks up the idempotency key in Redis. If found → return cached response immediately. No DB touched. If not found → continue.

Redis key: idempotency:{key} → cached PaymentResponse (TTL: 24h)
3

Validation

Validate: source wallet exists, destination wallet exists, amount > 0, source wallet has sufficient balance, merchant is active.

4

Create Payment Record (CREATED)

Insert a payment row with status = CREATED. This is the anchor record. All subsequent steps reference this paymentId.

5

Atomic DB Transaction Begins

A single PostgreSQL transaction now wraps steps 5–8. If anything fails, the entire transaction rolls back. No partial state.

@Transactional — Spring manages this. PostgreSQL guarantees ACID.
6

Lock Source Wallet (Pessimistic Lock)

Acquire a row-level lock on the source wallet using SELECT FOR UPDATE. Prevents concurrent payments from the same wallet from racing.

@Lock(LockModeType.PESSIMISTIC_WRITE) on wallet query
7

Debit Source Wallet / Credit Destination Wallet

Deduct amount from source wallet balance. Add amount to destination wallet balance. Both writes in the same transaction.

8

Create Ledger Entries (Double-Entry)

Two ledger rows are inserted: one DEBIT for the source account, one CREDIT for the destination account. Same amount. Same transactionId.

SUM(debits) == SUM(credits) always. Money cannot be created or destroyed.
9

Update Payment Status (SUCCESS)

Payment record is updated to status = SUCCESS. Timestamp recorded. All within the same transaction.

10

Transaction Commits

PostgreSQL commits the transaction. All five writes (debit wallet, credit wallet, 2 ledger entries, payment status) are durable together.

11

Store Result in Redis

PaymentResponse is stored in Redis under the idempotency key with a 24h TTL. Any retry of the same request gets this cached response.

12

Return Response to Client

HTTP 200 with paymentId, status: SUCCESS, amount, timestamp. Client receives confirmation.

05 · Refund Flow

Reversing a Payment

1

Client Requests Refund

POST /payments/:paymentId/refund — only allowed if payment status is SUCCESS.

2

Validate Refund Eligibility

Check payment status is SUCCESS. Check no existing refund for this payment. Check merchant wallet has sufficient balance.

3

Atomic Reversal Transaction

Debit merchant wallet. Credit user wallet. Insert two new ledger entries (reverse direction). Update payment status to REFUNDED. All in one transaction.

New ledger entries are created — the originals are never modified. Immutability.
4

Return Refund Confirmation

HTTP 200 with refundId, status: REFUNDED, original paymentId.

09 · Idempotency Strategy

Preventing Duplicate Payments

The Problem

Client sends payment request. Network times out. Client doesn't know if the payment succeeded. Client retries. Without protection, money is deducted twice.

The Solution

Client generates a UUID and sends it as the Idempotency-Key header. System checks Redis before processing. If key exists → return cached response. If not → process and cache the result. 

// Redis key structure
key:   "idempotency:{uuid}"
value: PaymentResponse (JSON)
TTL:   86400 // 24 hours

// Flow
if Redis.exists(key):
  return Redis.get(key)  // cached
else:
  result = processPayment()
  Redis.set(key, result, TTL)
  return result
Edge case handled: What if two concurrent requests arrive with the same key before either is cached? The idempotency_key UNIQUE constraint on the payments table acts as the final safety net — the DB will reject the second insert.
10 · Settlement Architecture

Delayed Merchant Fund Release

1

Payment Succeeds

On SUCCESS, a settlement record is created with status = PENDING and settle_at = now() + 1 day (T+1) or + 2 days (T+2).

2

Scheduler Runs

A Spring @Scheduled job runs every hour. It queries all PENDING settlements where settle_at <= now().

@Scheduled(cron = "0 0 * * * *") — runs at the top of every hour
3

Release Funds

For each eligible settlement: credit merchant wallet, create ledger entry, update settlement status to SETTLED. Atomic per settlement record.

Note on wallet balance during settlement: For simplicity, the merchant wallet is credited immediately on payment success in this implementation. Settlement tracking exists as a record-keeping and simulation layer. In a real system, funds would be held in a platform escrow wallet and released only on settlement.
11 · Failure Handling

What Happens When Things Go Wrong

Insufficient Balance
Validation step catches this before the transaction begins. HTTP 422 Unprocessable Entity returned. No DB writes occur.
DB Transaction Failure
If any step within the @Transactional block throws, Spring rolls back the entire transaction. Wallet and ledger are untouched.
Concurrent Payment (Same Wallet)
Pessimistic lock on the wallet row via SELECT FOR UPDATE. Second concurrent request waits. No double-spend possible.
Duplicate Request
Idempotency key in Redis returns cached response. If Redis is down, the UNIQUE constraint on idempotency_key in PostgreSQL rejects the duplicate.
Inactive Merchant
Merchant status is checked before payment processing. HTTP 400 Bad Request if merchant is INACTIVE.
Invalid Wallet IDs
FK constraints in PostgreSQL ensure referenced wallets must exist. Application layer validates before DB write anyway.
12 · Reconciliation

Verifying the Books Balance

Reconciliation answers one question: does the ledger balance? Every debit must have a matching credit. If the sum of all debits does not equal the sum of all credits, something went wrong. 

-- Reconciliation query (runs on GET /ledger/reconcile)

SELECT
  SUM(CASE WHEN entry_type = 'DEBIT'  THEN amount ELSE 0 END) AS total_debits,
  SUM(CASE WHEN entry_type = 'CREDIT' THEN amount ELSE 0 END) AS total_credits,
  SUM(CASE WHEN entry_type = 'DEBIT'  THEN amount ELSE -amount END) AS net_balance
FROM ledger_entries;

-- Expected: total_debits == total_credits, net_balance == 0
13 · Key Interview Concepts

What This Project Demonstrates

🔒
Pessimistic Locking
SELECT FOR UPDATE on wallet rows prevents race conditions during concurrent payments from the same source wallet.
⚛️
ACID Transactions
All financial writes wrapped in a single PostgreSQL transaction. Atomicity guaranteed — partial state is impossible.
🔑
Idempotency
Redis-backed idempotency keys with DB-level UNIQUE constraint as fallback. Safe retries are a first-class concern.
📒
Double-Entry Accounting
Every financial movement creates two ledger entries. The ledger is append-only and immutable. Source of truth for all balances.
🏗️
Modular Monolith
Clean package boundaries without the operational overhead of microservices. Easy to extract later if needed.
⏰
Settlement Scheduling
Demonstrates understanding of T+1/T+2 settlement cycles used in real payment networks and card rails.
🔄
Reconciliation
Ability to verify system integrity by summing ledger entries. A critical operation in any financial system.
🛡️
Fail-Safe Design
Multiple layers of protection: application validation → DB constraints → transaction rollback → idempotency cache.
